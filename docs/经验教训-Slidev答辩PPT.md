# 经验教训 — Slidev毕业答辩PPT制作

## 任务概述
根据论文内容和代码库，用Slidev基于Markdown制作毕业答辩PPT和讲稿。

## 关键经验

### 1. Slidev项目结构
- 只需要 `package.json` + `slides.md` 两个核心文件
- 图片引用使用相对路径 `../template/figures/xxx.png`
- `npm run dev` 启动开发服务器，支持热更新
- `npm run build` 构建静态站点，`npm run export` 导出PDF

### 2. 答辩PPT结构设计（10-15分钟）
- 封面 → 目录 → 绪论 → 技术综述 → 系统设计 → 系统实现 → 智能算法 → 测试结果 → 总结展望 → 致谢
- 约26页slides，每页约30秒
- 重点放在智能算法设计（占4-5页），这是论文核心创新点

### 3. Slidev Markdown语法要点
- `---` 分隔幻灯片
- `layout: section` 用于章节分隔页
- `layout: intro` 用于内容页
- `v-clicks` 实现逐项显示动画
- 支持LaTeX数学公式 `$f(n) = g(n) + \varepsilon \cdot h(n)$`
- 支持代码块、表格、grid布局
- `class: text-center` 居中文字

### 4. 讲稿编写
- 每页对应一段讲稿，标注预计时长
- 讲稿与PPT页面一一对应，方便练习
- 总时长控制在12-15分钟

### 5. 图片资源复用
- 直接引用论文项目中的截图 `template/figures/`
- 架构图、ER图、运行界面截图等都可直接复用
- 无需重新截图

## Prompt模式
- 用户要求"基于xx内容制作PPT/演示文稿" → 先全面收集素材(论文+代码) → 规划结构 → 编写slides.md + 讲稿 → 安装依赖并验证

## 第二轮补充（答辩审稿与反AIGC质疑）

### 6. 答辩材料最怕“说得太满”
- PPT 和讲稿最危险的不是表达不完整，而是**过度完整、过度顺滑、过度像标准答案**
- 一旦出现大量“创新点、显著提升、工程落地、全链路协同、避免幻觉”之类高密度判断句，评委很容易怀疑是 AIGC 包装
- **教训**：答辩稿应优先追求“可信”，而不是“华丽”

### 7. 研究现状和背景数字必须能追溯来源
- 类似“物流成本占GDP比重14.1%”“国外缺少LLM工程实践”这类表述，如果答辩时说不出来源，风险极高
- **教训**：凡是统计数字、国内外对比、研究不足判断，都要准备来源或把口径收缩为“在本文检索范围内”

### 8. 创新点要区分理论创新与工程改进
- 策略模式、运行时切换、容器化部署等内容更适合作为工程实现特点，不宜硬说成学术创新
- **教训**：答辩中应把“创新点”降级表述为“工作特点”“方案改进”或“工程集成贡献”

### 9. LLM相关内容必须严格划边界
- 只要提到 LLM，评委大概率会追问：是否必要、是否真的提升、失败怎么办、是否只是API调用
- **教训**：必须把 LLM 明确为可关闭、可降级、受约束的增强模块，并把没有充分实验支撑的扩展表述移出正文

### 10. 截图验证不等于实验结论
- 功能截图只能证明系统跑起来了，不能直接证明算法优越性
- **教训**：测试页要拆成“功能可用性验证 / 算法样例验证 / 部署验证”，避免用一堆勾号制造“全部证明完毕”的错觉

## 第三轮补充（Q&A备战与答辩辅导）

### 11. 答辩准备不能只靠“改稿”，必须产出速查材料
- 单纯把PPT和讲稿改稳还不够，用户真正临场需要的是“可快速扫一眼的Q&A速查清单”和“按时间线组织的准备手册”
- **教训**：凡是涉及答辩、汇报、面试等高压口头场景，除了正文材料，还应补一份 cheatsheet

### 12. Q&A材料要区分短答版与展开版
- 用户临场可能只有20秒，也可能被追问到50秒以上
- **教训**：高频问题应同时准备“短答版 + 展开版 + 回答策略”，这样既能速答，也能追问时稳住口径

### 13. 辅导手册要按时间线组织，而不是只堆建议
- 用户真正执行时更需要“第-3天做什么、第-2天做什么、当天做什么”这种结构
- **教训**：凡是“准备型”文档，都应优先按时间线、场景和检查清单组织

### 14. 答辩辅导的核心不是教用户更会包装，而是更会收口径
- 对会质疑AIGC的评委，最有效的策略不是增加术语，而是统一总定位、收紧表述、承认边界
- **教训**：答辩辅导文档应围绕“可信度管理”展开，而不是围绕“华丽表达”展开
