## 1. 角色介绍：物流世界的四大主角

想象这是一个大型物流主题 RPG 游戏，有四个职业：

| 角色 | 游戏职业 | 他们的日常 | 技能树 |
|---|---|---|---|
| **顾客** | 任务发布者 | 下单买买买，等快递 | 下单、支付、催快递、签收 |
| **商户** | 装备供应商 | 管理仓库、打包发货 | 库存管理、创建运单、发货 |
| **运输员** | 冒险执行者 | 接单、开车、送货 | 路径规划、状态更新、GPS 导航 |
| **管理员** | 世界管理者 | 调度中心、Hub 作业台 | MCMF 干线规划、K-Means 聚类、中转站管理 |

### 订单的奇幻漂流路线

**B2C 模式（最常见的冒险路线）**：
```
顾客下单 → 商户发货（分配 Hub）→ 入调度池 → 揽收入仓 → 入库 → 分拣
    ↓
MCMF 干线规划 → 出库 → 干线运输（发车/到达）→ 入目标城市调度池
    ↓
K-Means 末端聚类 → 创建配送批次 → 运输员承接 → 取件 → 配送 → 签收 ✅
```

**C2C 个人寄件（快捷通道）**：
```
顾客发起寄件 → 支付（自动分配 Hub）→ 入调度池 → ...（后续同上）
```

## 2. 序幕：顾客下单，冒险开始！

### 2.1 B2C 平台下单：我在网上买了东西

**场景**：你打开 APP，浏览商品，看中了一盒"优质坚果礼盒"，点击"立即下单"...

**前端页面**：`Customer/CreateOrder.vue` → 点击"提交订单"按钮

```javascript
// 你点击的那一瞬间，前端悄悄做了这些：
submitOrder()
  → POST /api/orders  { shopId, addressId, items, remark }
    → OrderController.createOrder()  // 后端接单啦
      → 检查：你真的是顾客吗？（roleCode 必须是 "customer"）
      → resolveCustomerId(userId)  // 把 userId 翻译成 customerId（Feign 调 customer-service）
      → orderService.createOrder(customerId, request)
        → 校验收货地址是不是你的
        → 校验商品信息和库存（Feign 调 shop-service）
        → 生成订单号："ORD" + 时间戳 + 4位随机  // 比如 ORD2026052012345678
        → 订单状态 = 1(待发货)，支付状态 = 1(已支付)  // 毕设简化：下单即支付
        → MQ 异步扣减库存: StockDeductMessage → ROUTING_STOCK_DEDUCT
  → 页面跳转到 '/sender/home/orders'（你可以在订单列表看到它了）
```

**订单诞生了！** 它现在状态是"待发货"，等着商户来发货。

### 2.2 C2C 个人寄件：我要寄个包裹

**场景**：你有东西要寄给朋友，点击"个人寄件"...

**前端页面**：`Customer/CreateShipment.vue` → 三步骤向导（取件地址 → 收件地址 → 货物信息）

```javascript
// 寄件三步走：
submitShipment()
  → POST /api/orders/personal-shipment  { senderAddressId, deliveryAddressId, cargoName, weight, ... }
    → OrderController.createPersonalShipment()
      → 检查：你真的是顾客吗？
      → resolveCustomerId(userId)
      → orderService.createPersonalShipment(customerId, request)
        → 查取件地址（要有坐标哦，不然 Hub 分配会降级）
        → 查收件地址
        → 运费计算：基础 12 元 + 超出 1kg 部分 2 元/kg
        → 订单状态 = 0(待支付)，支付状态 = 0(未支付)
        → 创建 OrderItem（货物名 + 申报价值）
  → 跳转到订单列表

// 然后你需要支付：
payOrder(orderId)
  → PUT /api/orders/{id}/pay
    → OrderServiceImpl.payOrder()
      → 个人寄件分支 (orderType=1):
        → logisticsFeignClient.assignHubs()  // 🎯 重点！分配 Hub
        → orderStatus = 2(待揽件)  // 跳过商户发货（因为是个人寄件嘛）
        → MQ: AddToPoolMessage → ROUTING_ADD_TO_POOL  // 入调度池
        → dispatchOriginType = 2(个人取件/上门揽收)
```

**关键差异**：
- B2C 下单：状态 0→1（待支付→待发货），需要商户发货
- C2C 寄件：支付后直接 2（待揽件），系统自动分配 Hub，进入物流流程

## 3. 第一幕：商户发货，Hub 初体验

**场景**：商户后台看到新订单，点击"发货"按钮，选择从哪个仓库发货...

**前端页面**：`Shop/Order.vue` → 点击"发货" → 弹出仓库选择对话框 → 确认

```javascript
// 商户视角：我有新订单啦，要发货！
handleConfirmShip()
  → POST /api/orders/{id}/ship  { warehouseId }
    → OrderController.shipOrder()
      → resolveShopId(userId)  // userId → shopId
      → orderService.shipOrder(orderId, shopId, warehouseId)
        → 校验：这订单真的是我的吗？状态是已支付且待发货吗？
        → 🎯 关键步骤：logisticsFeignClient.assignHubs()  // 分配 Hub！
          → LogisticsService.assignHubs(orderId, warehouseId)
            → 根据 warehouseId 查仓库坐标
            → 根据 endAddress 地理编码查收货坐标
            → 查找最近 Hub：按坐标距离排序（Haversine 公式）
            → 如果发货 Hub ≠ 收货 Hub → isCrossCity = 1（跨城啦！）
            → 写入 originHubId, destHubId
        → 更新订单: warehouseId + originHubId + destHubId + orderStatus = 2(待揽件)
        → 📬 MQ 消息：AddToPoolMessage → ROUTING_ADD_TO_POOL
          → LogisticsConsumer.handleAddToPool()
            → 写入 dispatch_pool 表
            → dispatchOriginType = 0(跨城揽收待处理)
        → 📦 自动创建出库单（失败也不影响发货主流程）：
          → shopFeignClient.createOutboundOrder()
          → shopFeignClient.completeOutboundOrder()  // 直接完成出库
```

### Hub 是个啥？

**Hub = 中转站**（物流分拨中心），就像快递的世界里有很多"驿站"：

```
商户仓库 → 发货 Hub（离仓库最近的） → [干线运输] → 收货 Hub（离顾客最近的） → 运输员 → 顾客
```

- **发货 Hub**（`originHubId`）：包裹的第一站，负责揽收和出库
- **收货 Hub**（`destHubId`）：包裹的最后一站，负责入库和末端派送

如果发货和收货分配到同一个 Hub（即 `crossCity=false`），那就是同城配送，不需要走干线运输，直接末端配送搞定！

## 4. 第二幕：中转站五重奏

**前端页面**：`Admin/HubOperations.vue`

这是一个**五阶段流水线看板**，管理员选择某个 Hub 后，可以逐阶段处理该 Hub 下的订单。想象这是一个物流版的"工厂流水线"！

### 页面初始化：加载 Hub 列表

```javascript
onMounted()
  → loadHubsWithOrders()
    → GET /api/logistics/national-network/hubs?level=2  // 所有城市配送中心
    → 对每个 Hub 并行: GET /api/logistics/dispatch/collection-queue?originHubId={id}
    → 过滤：只保留有待揽收订单的 Hub
    → 自动选中第一个 Hub → loadAll()
      → 并行加载五个阶段的数据
```

### Stage 0：待揽收 📦

**展示内容**：该 Hub 下所有**跨城已发货但尚未入干线的订单**（来自 `dispatch_pool` 表，`dispatch_origin_type=0`）。

| 操作 | 前端函数 | API 调用 | 后端发生了啥 |
|---|---|---|---|
| 确认已到仓（单条） | `doCreateInbound(item)` | `POST /logistics/hub-operations/inbound` | 创建 `hub_inbound_record`，状态 PENDING |
| 一键全部到仓 | `batchCreateInbound()` | 对每个 poolItem 调用 `createHubInboundRecord` | 逐条创建入库记录 |

**场景**：运输员小哥到了仓库，扫码确认"我取到货啦！"

### Stage 1：待入库 📥

**展示内容**：该 Hub 下所有**状态为 PENDING 的入库记录**（已经揽收了，等入库登记）。

| 操作 | 前端函数 | API 调用 | 后端发生了啥 |
|---|---|---|---|
| 确认入库 → 进入分拣 | `doConfirmInbound(rec)` | ① `PUT /logistics/hub-operations/inbound/{id}/confirm` → ② `POST /logistics/hub-operations/sorting` | ① 入库记录状态 PENDING→DONE，记录入库时间 ② 创建 `hub_sorting_record`，状态 PENDING |
| 手动登记 | `submitInbound()` | `POST /logistics/hub-operations/inbound` | 创建入库记录 |
| 全部入库 | `batchConfirmInbound()` | 对每条记录串行执行 确认+创建分拣 | 同上，逐条处理 |

**场景**：包裹到了 Hub，工作人员扫码入库，系统自动创建分拣任务。

### Stage 2：待分拣 🔀

**展示内容**：该 Hub 下所有**状态为 PENDING 的分拣记录**。

| 操作 | 前端函数 | API 调用 | 后端发生了啥 |
|---|---|---|---|
| 手动登记分拣 | `submitSorting()` | `POST /logistics/hub-operations/sorting` | 创建分拣记录（可指定 destHubId） |
| 前往全国干线规划 | — | — | 页面提示"前往全国干线规划，自动分配批次" |

> **💡 关键点**：分拣阶段实际上是通过 **MCMF 干线规划**来自动分配批次的，不是手动操作！这就像快递公司用算法决定"今天哪些包裹走哪条干线"，而不是人工一个个分配。

### Stage 3：待出库 📤

**展示内容**：该 Hub 下所有**状态为 PENDING 的出库记录**。

- **当前无操作按钮**，纯展示
- 出库记录在 **MCMF 规划创建干线批次时自动生成**

**场景**：分拣完了，包裹在出库区等着装车，等 MCMF 算法规划好干线运输方案后，自动生成出库记录。

### Stage 4：干线运输 🚛

**展示内容**：从该 Hub 发出的所有**干线运输批次**（来自 `inter_city_batch` 表）。

| 操作 | 前端函数 | API 调用 | 后端发生了啥 |
|---|---|---|---|
| 标记发车 | `doDepartBatch(id)` | `POST /logistics/dispatch/national/{id}/depart` | `interCityBatchService.onDeparture(batchId)` → 状态 CREATED→DEPARTED |
| 标记到达 | `doArriveBatch(id)` | `POST /logistics/dispatch/national/{id}/arrive` | `interCityBatchService.onArrival(batchId)` → 状态 DEPARTED→ARRIVED + 将批次内订单写入目标城市调度池 |
| 批量发车 | `batchTrunkAction('depart')` | 对每个 CREATED 批次调 departBatch | 同上，逐条 |
| 批量到达 | `batchTrunkAction('arrive')` | 对每个 DEPARTED 批次调 arriveBatch | 同上，逐条 |

**场景**：干线运输车装货完毕，点击"发车"；到了目的地 Hub，点击"到达"。

## 5. 第三幕：全国干线大调度（MCMF 算法登场！）

**前端页面**：`Admin/Dashboard.vue` 中的"全国干线"Tab

### MCMF 是啥？

**MCMF = 最小费用最大流算法**（Minimum Cost Maximum Flow）。

想象你是一个物流调度员，有 10 个 Hub，今天每个 Hub 有一堆包裹要发往其他 Hub。你要决定：
- 每条线路发多少货？（流量）
- 怎么发最省钱？（费用）

这就是一个**网络流问题**：Hub 是节点，Hub 之间的线路是边，每条边有容量（每天最多能运多少）和费用（运一单位要多少钱）。MCMF 算法帮你找到**在容量限制下，用最小总费用运输最多货物的方案**。

### 操作流程

```javascript
// 1. 点击"规划运力" 按钮
→ POST /api/logistics/dispatch/national/plan?planDate=...&llmReferenceDate=...
  → DispatchController.triggerNationalPlan()
    → flowPlanService.triggerManually(date, llmRef)
      → Step 1: 收集各 Hub 的供给量/需求量（从 dispatch_pool 汇总）
      → Step 2: LLM 费率校准（可选，调用 DeepSeek API 修正 HubLink 上的运输费用）
      → Step 3: 执行 MCMF 算法（单商品或多商品模式）
      → Step 4: 生成 FlowPlan + FlowPlanEdge（各边的流量分配）
      → Step 5: 创建 InterCityBatch（干线运输批次）
        → 创建 HubOutboundRecord（出库记录）
        → 更新 HubSortingRecord（分拣记录 → ASSIGNED）
        → 写入 dispatch_pool（目标端待调度条目）

// 2. 查看规划结果
→ GET /api/logistics/dispatch/national/latest-plan
  → 返回 FlowPlanDetailDTO（含各边流量、各 OD 对路由方案）

// 3. 发车/到达（在中转站看板操作，见上一章）
```

### MCMF 触发时发生了什么（超详细版）

```
flowPlanService.triggerManually(date, llmRef)
 ├── Step 1: 收集各 Hub 的供给量/需求量
 │     → 遍历 dispatch_pool，按 originHubId/destHubId 汇总
 │
 ├── Step 2: LLM 费率校准（可选）
 │     → LlmEdgeCostCalibrationServiceImpl.calibrate()
 │     → 调用 DeepSeek API，传入各 HubLink 的基本信息
 │     → 返回校准后的 costPerUnit
 │
 ├── Step 3: 执行 MCMF 算法
 │     → 单商品模式: McmfServiceImpl.computeMinCostFlow()
 │     → 多商品模式: McmfServiceImpl.computeMultiCommodityFlow()
 │
 ├── Step 4: 保存规划结果
 │     → 写 flow_plan 表
 │     → 写 flow_plan_edge 表（各边流量）
 │
 └── Step 5: 创建运输批次
       → 遍历有流量的边，创建 InterCityBatch
       → 对每条边上的订单：
         → 创建 HubOutboundRecord（出库记录）
         → 更新 HubSortingRecord.sortResult = ASSIGNED
         → 写入目标端 dispatch_pool（dispatchOriginType=1，跨城到达待调度）
```

## 6. 第四幕：末端配送 K-Means 派对

**前端页面**：`Admin/Dashboard.vue` 中的"末端调度"Tab

### K-Means 是啥？

**K-Means = K均值聚类算法**。

想象你是一个配送站长，手上有 100 个待配送订单，分散在城市各个地方。你有 5 个运输员空闲。你要决定：
- 怎么把 100 个订单分成 5 组？
- 每组分配给一个运输员，每组的订单要尽量靠近（减少配送路程）？

K-Means 帮你把地理上相近的订单聚成一簇，每簇分配给一个运输员。

### 操作流程

```javascript
// 1. 加载调度池
→ GET /api/logistics/dispatch/pool?destHubId=...
  → 返回该城市下所有待调度订单（同城单 + 跨城到达单）

// 2. 调度预览（K-Means 聚类 + 可选 LLM 建议）
→ GET /api/logistics/dispatch/preview?k=...&skipLlm=...
  → DispatchController.previewDispatch()
    ├── K-Means 聚类
    │     → clusterService.cluster(items, k)
    │     → KMeansClusterServiceImpl.cluster()
    │       → Haversine 距离计算（球面距离）
    │       → 均匀抽样初始化（不是随机选！保证初始中心均匀分布）
    │       → 迭代至收敛（最多 100 次）
    │       → 返回 List<ClusterResultDTO>
    │
    └── LLM 全局建议（skipLlm=false 时）
          → globalDispatchAdviceService.advise(clusters, warehouseId, lat, lng)
          → 调用 DeepSeek API，传入各簇的地理分布
          → 返回: 建议批次数、每簇是否经 Hub、紧急程度

// 3. 执行调度
→ POST /api/logistics/dispatch/execute
  → DispatchController.executeDispatch()
    → 对每个批次:
      → logisticsBatchService.createBatch(batchReq)
        → 创建 LogisticsBatch 记录
        → 创建 Delivery 记录（分配运输员）
        → 规划配送路线（A* 或 LLM 策略）
      → markDispatched(orderIds, batchId)
        → dispatch_pool 状态 0→1
```

## 7. 第五幕：运输员小哥的配送日常

**前端页面**：`Driver/Delivery.vue`

### 配送状态机

```
待承接(0) → 已接单(1) → 配送中(2) → 已送达(3)
                      ↘ 已取消(-1)
```

### 操作详解

| 操作 | API 调用 | 后端发生了啥 |
|---|---|---|
| **确认承接** | `POST /drivers/deliveries/{id}/accept` | 分配车辆，状态 0→1 |
| **开始配送** | `PUT /drivers/deliveries/{id}/status` + status=2 | 状态 1→2 |
| **确认完成** | `PUT /drivers/deliveries/{id}/status` + status=3 | 状态 2→3，更新订单状态 |
| **逐站送达** | `PUT /drivers/deliveries/{id}/complete-stop` | 多停靠点时，标记单个订单已送达 |
| **查看导航** | 跳转 `/driver/home/navigation?deliveryId=xxx` | 加载 Leaflet 地图 + 路线 GeoJSON |

### 路径规划：三种策略

运输员查看导航时，系统调用路径规划服务：

```java
RouteStrategy.plan(startLat, startLon, endLat, endLon, plannedTime)
```

三种策略可选（通过配置切换）：

| 策略 | 说明 | 适用场景 |
|---|---|---|
| **AStarRouteStrategy** | 纯 GraphHopper A* 路径规划 | 标准最短路线 |
| **LlmJudgeRouteStrategy** | A* 生成 3 条候选 + LLM 裁决最优 | 需要规避拥堵/选择更优路线 |
| **LlmWaypointRouteStrategy** | LLM 指定绕行路点 + A* 分段执行 | 需要绕路（比如要经过某个地标） |

## 8. 番外：智能算法天团介绍

### 8.1 MCMF 最小费用最大流（干线调度核心）

**文件**：`McmfServiceImpl.java`（456 行）

#### 单商品模式：SSP + SPFA

**核心思想**：将全国物流网络建模为流量网络，用 SSP（逐次最短路径）算法找到最小费用的流量分配方案。

**建图过程**：

```
1. 拆点：每个 Hub 拆成入港(in)和出港(out)两个节点
   - out(i) = 2*i,  in(i) = 2*i+1
   - 超级源 S = 2*hCount, 超级汇 T = 2*hCount + 1

2. 建边：
   - Hub 内换载边：in(i) → out(i)，容量 = max(总供给, 总需求)，费用 = 0
   - 实际运输边：out(i) → in(j)，容量 = link.capacityDaily，费用 = costPerUnit × 100
   - 超级源 → 各 Hub 出港，容量 = 该 Hub 的供给量
   - 各 Hub 入港 → 超级汇，容量 = 该 Hub 的需求量
   - 每条正向边配一条反向边（容量 0，费用为负）

3. SSP 主循环：
   while (SPFA 找到 S→T 的最短费用路径) {
     沿路径增广（推送瓶颈流量）
     更新残差网络
   }

4. 收集结果：遍历原始边，flow = 原始容量 - 残余容量
```

**为什么费用乘 100？** 避免浮点误差——所有费用取整后用 `long[]` 存储，最终结果除以 100 还原为 `BigDecimal`。

#### 多商品模式：Dijkstra-SSP

**与单商品的本质区别**：每个 OD 对（Origin-Destination）有独立的供需，不会出现"同一 Hub 供给与需求代数抵消"的建模缺陷。

```
1. 构建残差图：costAdj（费用邻接表）+ resCap（剩余容量，跨商品共享）

2. 按 OD 需求量降序处理（高需求优先占容量）：
   for each OD pair:
     while (该 OD 还有剩余需求):
       dijkstraOnResidual(originHub, destHub)  // 找最短费用路径
       bottleneck = 路径上最小残差
       增广 min(bottleneck, remaining) 单位流量
       更新共享残差容量

3. 汇总各边流量
```

**Dijkstra 实现**：标准优先队列 Dijkstra，O((V+E)logV)，只走残差 > 0 的边。

### 8.2 K-Means 末端聚类（末端调度核心）

**文件**：`KMeansClusterServiceImpl.java`（110 行）

#### 算法流程

```
1. 确定聚类数 k
   - 若未指定，默认 k = ceil(sqrt(n))
   - k = min(k, n)

2. 均匀抽样初始化
   - 将 n 个订单按某种顺序排列
   - 均匀抽取 k 个作为初始中心：
     idx = round(i / (k-1) * (n-1))  (i=0,1,...,k-1)
   - 不是随机选，保证初始中心在数据分布上均匀覆盖

3. 迭代（最多 100 次）：
   E-step: 每个订单分配到 Haversine 距离最近的中心
   M-step: 重新计算各簇中心的经纬度算术平均
   收敛条件: 无任何订单改变簇归属 → 提前终止

4. 输出：List<ClusterResultDTO>，每个簇含中心坐标、成员订单列表
```

#### Haversine 距离

```java
// 地球半径 6371 km
// 标准球面距离公式
d = 2R × arcsin(√(sin²((lat2-lat1)/2) + cos(lat1)×cos(lat2)×sin²((lon2-lon1)/2)))
```

> **注意**：簇中心重计算用的是经纬度算术平均，在小区域内误差可接受（城市级别）。

### 8.3 A* 路径规划（配送路线核心）

**文件**：`AStarRouteStrategy.java`

#### 基础 A*

```
1. 经纬度 → GraphHopper LocationIndex → 最近路网节点
2. A* 搜索：
   - openSet: PriorityQueue，按 f(n) = g(n) + h(n) 升序
   - g(n): 起点到当前节点的实际距离
   - h(n): Haversine(当前节点, 终点)，可采纳的启发函数
   - 每次取 f 最小的节点扩展
3. 重建路径 → 输出 GeoJSON LineString
```

#### Weighted A*（多候选路线生成）

通过调整 `epsilon` 参数生成不同的备选路线：

| epsilon | 效果 | 用途 |
|---|---|---|
| 1.0 | 标准最优 A* | 最短路线 |
| 1.8 | 膨胀启发值，更激进趋向终点 | 较快但可能次优 |
| 3.5 | 极度膨胀 | 明显不同的备选路线 |

```
h(n) = epsilon × Haversine(当前节点, 终点)
```

epsilon > 1 时不再保证最优，但能快速生成不同路线供 LLM 裁决。

### 8.4 LLM 裁决策略（智能路线选择）

**文件**：`LlmJudgeRouteStrategy.java`

```
1. 对同一 OD 调用三次 A*（epsilon=1.0, 1.8, 3.5），生成三条候选路线
2. 构造 Prompt：
   - 三条路线距离、预计时间、途经道路类型
   - 要求 LLM 返回 JSON: { recommended: 1|2|3, reason: "..." }
3. 调用 DeepSeek API
4. 解析 JSON，返回推荐路线
5. 降级容错：如果 LLM 返回异常，回退到 epsilon=1.0 的最优路线
```

### 8.5 LLM 路点策略（智能绕路）

**文件**：`LlmWaypointRouteStrategy.java`

```
1. 将起点、终点、途经关键点信息发给 LLM
2. LLM 决定绕行路点（战略层）: "建议经过 XX 路/YY 桥"
3. A* 分段执行（战术层）: 起点→路点→终点
4. 拼接多段 GeoJSON
```

### 8.6 LLM 费率校准（智能费用修正）

**文件**：`LlmEdgeCostCalibrationServiceImpl.java`

```
1. 收集各 HubLink 的基本信息（距离、城市、货物类型）
2. 构造 Prompt：要求 LLM 根据当前市场情况修正运输费用
3. DeepSeek API 返回各边的校准后费率
4. 更新内存中的 costPerUnit（不修改数据库）
```

### 8.7 LLM 全局调度建议（智能调度辅助）

**文件**：`LlmFlowPlanAdviceServiceImpl.java`

```
1. 输入：K-Means 聚类结果 + 仓库坐标
2. 构造 Prompt：各簇的地理分布、订单量、距仓库距离
3. DeepSeek API 返回：
   - 建议批次数
   - 每簇是否经 Hub 中转
   - 紧急程度判断
4. 与 K-Means 结果合并，返回 DispatchPreviewDTO
```
