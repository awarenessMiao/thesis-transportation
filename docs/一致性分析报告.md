## 一、严重矛盾（答辩时被问会直接穿帮）

### 1.1 K-Means 的 k 值取法：论文 vs Slides/讲稿 vs 代码 三方不一致

| 来源 | 调度预览 k 值 | 末端分组 k 值 |
|------|-------------|-------------|
| **论文** | **管理员手工指定** | **未区分两处实现**（只描述了一个K-Means场景） |
| **Slides** | `⌈√n⌉` | `⌈N/4⌉`（每司机≤4停靠点） |
| **讲稿** | `⌈√n⌉`（"工程启发式取值，不是理论最优"） | 已提及末端分组 |
| **代码 KMeansClusterServiceImpl** | `k=null` 时默认 `⌈√n⌉` | — |
| **代码 LogisticsBatchServiceImpl** | — | `k = ⌈N/4⌉`（`Math.ceil(n / 4.0)`） |

**矛盾分析**：
- 论文明确写"k值由管理员指定"，但代码中当k=null时默认取⌈√n⌉，Slides/讲稿写⌈√n⌉为默认取值
- 论文没有区分两处 K-Means 实现，但代码中确实有两个不同的 K-Means 实现（`KMeansClusterServiceImpl` 和 `LogisticsBatchServiceImpl.kMeansGroup()`），且 k 值不同
- **对齐方案**：论文中加一句"当管理员未指定k值时，系统默认取⌈√n⌉"；同时在实现章节补充末端分组的第二处 K-Means 描述（k=⌈N/4⌉）

### 1.2 K-Means 最大迭代次数：论文/Slides vs 代码不一致

| 来源 | 最大迭代次数 |
|------|------------|
| **论文** | 100 次 |
| **Slides/讲稿** | 100 次 |
| **代码 KMeansClusterServiceImpl** | 100 次 ✅ |
| **代码 LogisticsBatchServiceImpl** | **50 次** ❌ |

**矛盾**：末端分组的 K-Means（`LogisticsBatchServiceImpl.KMEANS_MAX_ITER = 50`）迭代上限是 50，而非论文和 Slides 中说的 100。

**对齐方案**：Slides/讲稿中提到"两处实现"时补充说明末端分组迭代上限为50次；论文参数配置表中也应区分两处实现的参数差异

### 1.3 K-Means 初始化方式：论文/Slides vs 代码不一致

| 来源 | 初始化方式 |
|------|----------|
| **论文** | **K-Means++** |
| **Slides/讲稿** | **K-Means++** |
| **代码 KMeansClusterServiceImpl** | **均匀抽样**（等间距索引采样，非 K-Means++） |
| **代码 LogisticsBatchServiceImpl** | **均匀抽样**（同 KMeansClusterServiceImpl，等间距索引采样） |

**矛盾**：论文和 Slides 都写了使用 K-Means++，但代码中**两处 K-Means 实现均使用均匀抽样初始化**（按索引等间距抽取中心点），并非 K-Means++。

**代码实际实现**（KMeansClusterServiceImpl L39-46）：
```java
for (int i = 0; i < k; i++) {
    int idx = (int) Math.round((double) i / (k - 1 == 0 ? 1 : k - 1) * (n - 1));
    centerLat[i] = valid.get(idx).getEndLat();
    centerLng[i] = valid.get(idx).getEndLng();
}
```

**对齐方案**：将论文/Slides/讲稿中的"K-Means++初始化"修改为"均匀抽样初始化"，并说明选择理由（减少随机性，结果可复现）

### 1.4 MCMF 伪代码：论文实现章节 vs 设计章节不一致
