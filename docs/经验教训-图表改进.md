# 经验教训：论文图表 Mermaid/Fletcher 改进

## 日期
2026-05-17

## 任务
将 Transportation 毕业论文中的静态 PNG 图表替换为 Typst 原生矢量图（Mermaid / Fletcher），提升视觉效果和可维护性。

## 关键决策与教训

### 1. 工具选择：Mermaid vs Fletcher vs 原PNG
- **Mermaid**（通过 mmdr 包）最适合：标准流程图（3~5节点）、时序图。节点多时自动布局失控。
- **Fletcher** 最适合：简单状态机、转换图、数学架构图。复杂分层架构图代码量过大且难达专业效果。
- **原PNG** 最适合：复杂分层架构图、带泳道/角色的流程图、复杂ER图（>10实体）、有颜色分组需求的图。
- **核心原则**：矢量化的收益（清晰度+版本控制）必须大于视觉效果下降的损失。对于已经用专业工具精心绘制的图，不要盲目替换。

### 2. 包选择：mmdr vs beautiful-mermaid
- 选用 **mmdr:0.2.2**（基于 Rust 的 mermaid-rs-renderer），而非 beautiful-mermaid（基于 WASM + quickjs）。
- 理由：mmdr 编译更快（500-1000×），无浏览器依赖，且支持 flowchart / ER diagram 等核心类型。
- beautiful-mermaid 存在已知 CSS 变量 SVG 渲染 bug，不适合生产环境。

### 3. 长宽比控制（致命教训）
- Mermaid LR（横向）在A4页面中：节点>5个时，每个节点被极度压缩，整体扁平如一条线。
- Mermaid TD/TB（纵向）在A4页面中：节点>7个时，整体细长如一根筷子。
- Fletcher 单列布局：5个节点以上即呈现极端高宽比。
- **修复前必须先预估**：根据节点数量和页面宽度，预判最终高宽比是否合理。

### 4. Fletcher 架构图限制
- **无内置分组/泳道**：enclose 功能与专业绘图工具的分组效果差距甚远，无法复现颜色分层背景。
- **无特殊形状**：不支持云形、圆形、菱形等UML标准形状，只有rect/ellipse/pill等基础形状。
- **边优化困难**：手动坐标定位下，多层架构图的边极易交叉，需要精心设计的布局算法才能避免。

### 5. 中文字符串转义
- Mermaid 代码嵌入 Typst 字符串时，`|` 和 `-->` 无需转义，正常书写即可。
- 多行字符串在 Typst 中直接支持换行，无需 `\n`。

### 6. 编译验证策略
- `typst compile --root . template/thesis.typ output.pdf` 必须指定 `--root`，否则 `../lib.typ` 访问被拒绝。
- 字体警告（如 `stsongti` 缺失）不影响编译，可忽略。
- **验证图表不能只检查文本**：Mermaid/Fletcher可能编译通过且文本存在，但视觉布局可能一团糟。必须**实际查看PDF渲染效果**。
- 提取页面截图验证：`python3 + fitz` 提取页面为PNG，检查长宽比和节点分布。

### 7. 备份与回滚策略
- 被替换的 PNG **必须**备份到 `figures/backup/`，因为矢量化方案很可能需要回滚。
- 回滚不是失败，而是对视觉质量的正确判断。

### 8. 诚实评估原图质量
- 在决定替换前，必须**仔细查看原PNG的实际效果**。如果原图是用 ProcessOn/draw.io/Visio 等专业工具绘制的，效果通常远超Mermaid/Fletcher的默认输出。
- 分辨率低（如501×321）不应成为替换理由，如果替换后的矢量图视觉效果更差。

## 通用模式提取

此任务的 Prompt 通用模式可抽象为：
> "针对 [文档类型] 中的 [内容类型]，分析是否使用 [工具A/工具B] 重绘效果更好。先准备改进方案和升级对应关系，然后执行修改。可搜索 [技术领域] 在 [平台] 中的最佳实践。"

适用场景：
- 论文图表优化（Typst/Mermaid/Fletcher/Cetz）
- PPT 图表矢量化的工具选型
- 技术文档中的架构图/流程图升级
