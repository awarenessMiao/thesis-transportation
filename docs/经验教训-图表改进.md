# 经验教训：论文图表 Mermaid/Fletcher 改进

## 日期
2026-05-17

## 任务
将 Transportation 毕业论文中的静态 PNG 图表替换为 Typst 原生矢量图（Mermaid / Fletcher），提升视觉效果和可维护性。

## 关键决策与教训

### 1. 工具选择：Mermaid vs Fletcher vs 原PNG
- **Mermaid**（通过 mmdr 包）最适合：标准流程图（3~5节点）、时序图。节点多时自动布局失控。
- **Fletcher** 最适合：简单状态机、转换图、数学架构图。复杂分层架构图代码量过大且难达专业效果。
- **原PNG** 最适合：复杂分层架构图、带泳道/角色的流程图、复杂ER图（>10实体）、有颜色分组需求的图。
- **核心原则**：矢量化的收益（清晰度+版本控制）必须大于视觉效果下降的损失。对于已经用专业工具精心绘制的图，不要盲目替换。

### 2. 包选择：mmdr vs beautiful-mermaid
- 选用 **mmdr:0.2.2**（基于 Rust 的 mermaid-rs-renderer），而非 beautiful-mermaid（基于 WASM + quickjs）。
- 理由：mmdr 编译更快（500-1000×），无浏览器依赖，且支持 flowchart / ER diagram 等核心类型。
- beautiful-mermaid 存在已知 CSS 变量 SVG 渲染 bug，不适合生产环境。

### 3. 长宽比控制（致命教训）
- Mermaid LR（横向）在A4页面中：节点>5个时，每个节点被极度压缩，整体扁平如一条线。
- Mermaid TD/TB（纵向）在A4页面中：节点>7个时，整体细长如一根筷子。
- Fletcher 单列布局：5个节点以上即呈现极端高宽比。
- **修复前必须先预估**：根据节点数量和页面宽度，预判最终高宽比是否合理。

### 4. Fletcher 架构图限制
- **无内置分组/泳道**：enclose 功能与专业绘图工具的分组效果差距甚远，无法复现颜色分层背景。
- **无特殊形状**：不支持云形、圆形、菱形等UML标准形状，只有rect/ellipse/pill等基础形状。
- **边优化困难**：手动坐标定位下，多层架构图的边极易交叉，需要精心设计的布局算法才能避免。

### 5. 中文字符串转义
- Mermaid 代码嵌入 Typst 字符串时，`|` 和 `-->` 无需转义，正常书写即可。
- 多行字符串在 Typst 中直接支持换行，无需 `\n`。

### 6. 编译验证策略
- `typst compile --root . template/thesis.typ output.pdf` 必须指定 `--root`，否则 `../lib.typ` 访问被拒绝。
- 字体警告（如 `stsongti` 缺失）不影响编译，可忽略。
- **验证图表不能只检查文本**：Mermaid/Fletcher可能编译通过且文本存在，但视觉布局可能一团糟。必须**实际查看PDF渲染效果**。
- 提取页面截图验证：`python3 + fitz` 提取页面为PNG，检查长宽比和节点分布。

### 7. 备份与回滚策略
- 被替换的 PNG **必须**备份到 `figures/backup/`，因为矢量化方案很可能需要回滚。
- 回滚不是失败，而是对视觉质量的正确判断。

### 8. 诚实评估原图质量
- 在决定替换前，必须**仔细查看原PNG的实际效果**。如果原图是用 ProcessOn/draw.io/Visio 等专业工具绘制的，效果通常远超Mermaid/Fletcher的默认输出。
- 分辨率低（如501×321）不应成为替换理由，如果替换后的矢量图视觉效果更差。

### 9. 复杂业务流程图更适合“脚本绘图 + PNG 引用”
- 对于“多阶段 + 多泳道 + 多种交互方式标签”的流程图，直接在 `Typst` 里用 `diagram/fletcher` 拼节点，虽然代码短，但最终很容易退化成“只有一串节点的抽象示意图”，信息分层不足。
- 本次“系统核心业务流程图”改为使用 `PIL` 脚本生成：上方按业务阶段分列，左侧按参与角色/订单服务/物流服务/运输员服务分泳道，并显式标出 `Feign同步`、`MQ异步`、`管理员确认` 等关键交互。
- **教训**：当图的核心价值在于“版式组织能力”而不是“矢量源码优雅”时，应优先选择可精确控版的外部脚本或专业绘图工具。

### 10. 图稿接入不能只改源码，还要验证资源落点
- 本次重绘不仅更新了 `template/thesis.typ` 的图引用，还把生成的 `core-business-flow.png` 同步输出到 `template/figures/` 和 `slides/public/figures/`。
- **教训**：图稿若同时服务论文和答辩材料，应统一资源出口，避免论文和PPT目录各自维护一份，后期容易版本漂移。

### 11. 回编成功不代表图就一定对，需要额外核对输出文件
- 本次中途出现过“代码已改但 PNG 未落盘、Typst 编译时找不到新图”的问题。
- **教训**：复杂图稿改造必须采用“三步确认”：脚本成功执行 → 目标目录出现新文件 → 主文档成功引用并编译。

### 12. 箭头不能用“线段末端直接贴三角形”粗暴处理
- 如果箭头主干直接画到终点，再把三角形头部压在末端，视觉上很容易出现“箭头尖端发虚、根部发胖、与节点边框衔接不准”的问题。
- 本次修复改为：**主干线终止于箭头底边中心，箭头尖端单独精确落在连接锚点**，并给折线中间转角补圆点过渡。
- **教训**：流程图箭头应拆分为“折线路径 + 箭头头部”两个几何对象分别控制，不能只靠一条线和一个三角形碰运气。

### 13. 结构边框要统一成有限的描边令牌
- 本次将中性结构描边收敛为少量固定值：`BORDER = #98a1ab`、`GRID = #d4dbe3`、`LINE = #747d87`、`NOTE_BORDER = #d9e0e7`、`MUTED = #66707a`。
- 线宽也统一为：外框 `3px`、分隔与卡片描边 `2px`、主流程箭头 `4px`、箭头长度 `18px`、箭头半宽 `8px`。
- **教训**：复杂图稿不能让 `#8b8b8b / #7a7a7a / #90959d / #d9d9d9` 这类近似色自由散落，否则即使布局没错，也会显得脏和虚。

### 14. 固定像素布局会破坏多尺寸下的一致性
- 原图中大量 `18 / 24 / 26 / 38` 这类固定像素偏移，在缩小到答辩页或预览图时容易导致标签距离、箭头长度和圆角比例失衡。
- 本次为 `draw_core_business_flow()` 增加 `size` 参数与统一缩放函数 `u()`，使边距、圆角、字号、箭头尺寸按比例缩放。
- **教训**：如果一张图同时要进论文、PPT、网页预览，就应尽早把几何参数抽象成可缩放变量，而不是把默认分辨率当作唯一真相。

### 15. 用户嫌图“乱”时，优先删除冗余区，而不是继续堆说明
- 本次用户明确指出底部 `Feign同步 / MQ异步 / 人工确认 + 说明文字` 没和上面对齐，且整体显得更乱。最终最有效的处理不是重排，而是**直接删除整块底部 legend/note 区**。
- **教训**：如果箭头附近已经有就地标签，底部再放一套图例和说明通常只会形成重复信息；对于论文插图，应优先让图本身自解释，而不是在图内再塞一段小字说明。

### 16. 给复杂履约链路“让走廊”，比美化箭头更重要
- 用户本轮指出的核心问题不是箭头造型，而是 `Hub-and-Spoke` 链路中间盒子太宽、阶段间空隙太窄，导致箭头和标签没有呼吸空间。
- 本次修复采用了**非对称左右边距**：不是所有节点都等宽居中，而是有意识地缩窄 `K-Means聚类预览`、`生成logistics batch`、`到站后激活末端配送任务`、`接收末端任务`、`末端配送` 等盒子，为箭头让出专门走廊。
- **教训**：流程图如果要表达“链路”，应把留白优先分配给连接关系，而不是平均分配给节点本体。

### 17. 短而挤的箭头，不如直接合并为流程块
- 用户进一步指出：有些箭头起终点即使勉强对齐，只要距离过短、标签又必须贴近，就还是会显得脏。
- 本次最终处理不是继续微调短箭头，而是**直接把 `dispatch_pool + K-Means预览` 合并为一个 block，把 `接收任务 + 末端配送` 合并为一个 block**。
- **教训**：如果两个动作在论文层面不需要被单独强调，且它们之间的箭头已经短到只剩“连接感”，那就应该优先合并成一个更大的过程块。

### 18. 内嵌箭头标签应少而精，重叠时宁可删除
- `MQ异步`、`MQ激活` 这类说明如果放进拥挤区域，很容易与节点文本、边框和连线相互遮挡。
- 本次最终版本保留最有辨识度且有空间承载的 `Feign同步`，去掉了易重叠的其余内嵌标签，并把具体调用关系留给正文图注说明。
- **教训**：流程图中的文字优先级应是“节点语义 > 主链路方向 > 补充说明标签”；当空间冲突时，最该先删的是补充说明标签。

### 19. 论文风格流程图应优先使用正交走线
- 对于分阶段泳道图，斜线或带轻微偏移的折线很容易产生“像编辑器草稿”的观感。
- 本次末端激活与签收闭环相关连线最终改为**水平 + 垂直的正交走线**，减少了随意感。
- **教训**：当目标是论文插图而非产品流程图原稿时，应优先选择规则、克制、易预测的几何路径。

## 通用模式提取

此任务的 Prompt 通用模式可抽象为：
> "针对 [文档类型] 中的 [内容类型]，分析是否使用 [工具A/工具B] 重绘效果更好。先准备改进方案和升级对应关系，然后执行修改。可搜索 [技术领域] 在 [平台] 中的最佳实践。"

适用场景：
- 论文图表优化（Typst/Mermaid/Fletcher/Cetz）
- PPT 图表矢量化的工具选型
- 技术文档中的架构图/流程图升级

### 20. 用户要求“去颜色”时，要按黑白稿逻辑重建层次，而不是机械去饱和
- 本次用户要求“把颜色去掉，只留下黑白轮廓和边框”，如果只是把原色改成浅灰，泳道底色和阶段色块仍会形成大面积灰面，视觉上依然脏。
- 最终更有效的做法是收敛成三层令牌：**主轮廓 `#111111`、辅助分隔 `#8c8c8c`、面填充 `#ffffff`**，同时把阶段块、泳道块、流程块全部改成白底黑边。
- **教训**：论文图、打印稿、投影稿里的“去颜色”不等于简单灰度化，而是要重建一套单色层次系统。

### 21. 长箭头的指向问题，优先通过“同轴入射”解决
- 本次真正改善观感的关键，不是把箭头头部再缩几像素，而是让几条主链路都尽量改成**沿同一 `x` 或 `y` 轴进入目标块**：能垂直直入就不要先横后竖，能水平直入就不要多拐一次。
- 为此补了 `anchor_at_x()` / `anchor_at_y()`，根据目标框的坐标反推起止锚点，使 `调度池预览 -> 管理员确认`、`到站激活 -> 末端配送`、`末端配送 -> 签收完成` 等箭头变成更自然的直入路径。
- 后续微调中，`LogisticsService -> 写入调度池并进行 K-Means 聚类预览` 这条线又进一步改成**先向右再向下**，说明局部视觉最优路径不一定和全局策略完全一致。
- **教训**：用户说“箭头指向还有问题”时，优先检查的不是三角箭头大小，而是线段末端是否以最直接、最顺眼的坐标轴路径进入目标节点。
