## 关键经验

### 1. 论文架构：设计应先于实现
- Ch4"系统实现"有算法实现描述，但Ch3"系统设计"完全没有对应的设计描述
- **教训**：论文的"设计"章节必须有"算法设计"子节，这是独立于DB/API设计的另一个维度
- 正确的Ch3结构应该是：模块设计（DB+API）→ 算法设计 → 前端设计 → 数据库设计

### 2. 代码→论文的反向映射
- 代码中有6+种算法实现（A*、LLM Judge、LLM Waypoint、K-Means、MCMF单/多商品、策略模式），但论文只描述了4种
- **LlmWaypointRouteStrategy 完全未出现在论文中**——这是最复杂的算法之一
- **教训**：每次写论文前应先做代码审计，列出所有算法实现与论文描述的映射关系

### 3. 策略模式是算法架构的核心
- `RouteStrategy`接口 + 多实现 + Spring配置切换，这个架构设计本身就是论文应该描述的重要内容
- **教训**：设计模式的使用应该在"系统设计"章节中描述，而不仅是在"实现"章节中提及

### 4. 数据库种子数据问题
- `mcmf_test_data.sql` 存在但未加入docker-compose.yml自动加载
- `dispatch_pool` 为空导致K-Means预览完全无法使用
- **教训**：所有测试数据脚本都应加入Docker初始化流程，确保`docker compose up -d`后系统即可演示
- 创建了 `local_dispatch_seed.sql` 补充上海同城待调度订单数据

### 5. Typst数学模式中的希腊字母
- `$epsilon$` 在Typst数学模式中被解析为 `e` + `psilon` 两个变量，不会渲染为ε
- **正确写法**：直接使用Unicode字符ε，如 `$ε > 1$` 或行内 `ε=1.0`
- **教训**：Typst数学模式中，优先使用Unicode希腊字母而非LaTeX风格的转义

### 6. Fletcher图表中的括号转义
- Fletcher的 `dg-rect` 中 `(Auth)` 被Typst解析为函数调用
- **正确写法**：`\\(Auth\\)` 需要转义括号
- **教训**：Fletcher/Typst中所有非函数调用的括号都必须转义

## 第二轮补充（截图重拍）

### 7. Docker MySQL初始化的collation冲突
- `mcmf_test_data.sql` 和 `local_dispatch_seed.sql` 中的 `LIKE CONCAT(...)` 查询会触发 `utf8mb4_0900_ai_ci` vs `utf8mb4_unicode_ci` 的collation冲突
- **修复方法**：在LIKE子句后加 `COLLATE utf8mb4_unicode_ci`
- **教训**：Docker MySQL的默认collation与Spring Boot应用的连接collation可能不同，种子数据SQL中的字符串比较需要显式指定COLLATE
- **03和04的SQL因collation冲突无法自动加载**，改为手动执行：`docker exec -i transport-mysql bash -c 'mysql -u root -prootpassword transportation' < database/mcmf_test_data.sql`

### 8. Docker服务启动顺序
- Nacos未就绪时启动微服务会导致注册失败（Client not connected, current status:STARTING）
- **正确做法**：先启动基础设施（mysql/redis/rabbitmq/nacos）→ 等待healthy → 再启动业务服务
- `docker compose up -d` 默认并行启动，需通过 `depends_on` + healthcheck 或分步启动确保顺序

### 9. Playwright CLI自动化截图
- 使用 `playwright-cli` 可快速完成多角色登录→导航→截图的自动化流程
- **snapshot验证**比OCR更可靠：通过DOM树直接验证页面内容（标题、按钮、数据量等）
- **截图分辨率**：`playwright-cli resize 1440 900` 设置为标准桌面分辨率
- **多角色切换**：`localStorage.clear()` + `page.goto(login)` 即可切换角色
- 中文OCR（tesseract）需要额外安装 `chi_sim` 语言包，但snapshot验证已足够

### 10. 截图内容验证要点
- 管理员仪表板：显示"93待揽件订单"（dispatch_pool总量）
- 末端调度预览：K-Means将25单分为5批次，LLM分析建议（直达vs经分拨中心）
- 运力规划：MCMF分配520件、¥4430.8
- 运输员：1单进行中任务
- **教训**：截图前应先确认数据已正确加载到所有相关表

## 第五轮补充（代码对齐核查）

### 18. K-Means初始化方式：论文写了K-Means++，代码实际是均匀抽样
- 论文/Slides/讲稿中描述K-Means初始化为"K-Means++"，但代码验证发现**两处实现都是均匀抽样**（按等间距索引选取初始中心）
- **KMeansClusterServiceImpl L39-46**：`int idx = (int) Math.round((double) i / (k - 1 == 0 ? 1 : k - 1) * (n - 1));`
- **LogisticsBatchServiceImpl L620-627**：同上，均匀抽样
- **修复**：论文/Slides/讲稿中"K-Means++初始化"全部改为"均匀抽样初始化"
- **教训**：论文描述算法细节前，必须先读代码确认实际实现方式，不能凭理论假设填写

### 19. K-Means收敛条件：论文写了0.001°阈值，代码实际只检查分配变化
- 论文/Slides/讲稿中描述收敛条件为"中心偏移<0.001°或100次迭代"，但代码验证发现**两处实现都只检查assignments是否变化**

- 没有0.001°阈值判断
- **修复**：论文/Slides/讲稿中收敛条件改为"分配不变或达到最大迭代次数"
- **教训**：收敛条件描述应以代码实际逻辑为准，"分配不变"是比"中心偏移"更直接也更准确的描述

### 20. 数据库表数量：Slides写28张，实际38张

## 第六轮补充（Slides渲染问题修复）

### 24. Slidev中`**`加粗标记在HTML内不渲染
- Slidev中，`**text**`语法在`<div>`等HTML标签内可能不被Markdown处理器识别，导致渲染后`**`原样显示
- **修复**：HTML标签内的`**text**`全部替换为`<strong>text</strong>`
- **教训**：Slidev的Markdown处理与标准Markdown有差异，HTML标签内应使用HTML标签而非Markdown语法

### 25. Slidev中LaTeX公式不渲染
- Slidev默认不支持KaTeX/LaTeX语法，`$f(n) = ...$`会原样显示
- **修复**：将简单LaTeX公式转为Unicode符号表示，如`$f(n) = g(n) + \varepsilon \cdot h(n)$` → `f(n) = g(n) + ε·h(n)`
- **教训**：对于演示文稿，Unicode数学符号比LaTeX更可靠；如需完整LaTeX支持需安装`markdown-it-katex`插件

### 26. 浏览器自动翻译导致中文内容乱码
- 中文Slidev演示在Chrome等浏览器中可能被自动翻译，导致：
  - "LLM" → "法学硕士"（Large Language Model vs Master of Laws的歧义）
  - "Hub" → "皇冠"（Hub在英语中有crown/hub的歧义）
  - "绪论" → "论绪"（翻译引擎重排）
  - "技术选型总览" → 被翻译为奇怪的英文表述
- **修复**：
  1. frontmatter中添加`lang: zh-CN`，明确文档语言
  2. 关键缩写首次出现时用中文全称+英文缩写，如"大语言模型(LLM)"
  3. 保留"Hub"原文不翻译，避免被翻译引擎误译
- **教训**：任何面向浏览器渲染的中文内容，都应设置正确的`lang`属性，并用"中文全称(英文缩写)"格式防止自动翻译

### 27. "Agent"在学术语境中的翻译
- "Agent自主决策与工具调用"中的"Agent"在学术中文中应翻译为"智能体"，不是"代理"
- **修复**：未来展望中"Agent"改为"智能体"
- **教训**：AI领域的"Agent"在学术论文/答辩中应使用"智能体"而非"代理"

### 28. Slidev目录页遮挡后续内容
- 使用`layout: intro`的目录页在Slidev中可能与导航控件重叠或遮挡后续slide内容
- **修复**：删除独立目录页，依靠`layout: section`的分节页作为导航
- **教训**：Slidev的分节页（section layout）本身就起到目录导航作用，额外目录页既冗余又可能造成遮挡
