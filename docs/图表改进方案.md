## 二、Typst 学术绘图工具对比

| 工具 | 类型 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|----------|
| **Mermaid** (beautiful-mermaid/mmdr) | 声明式文本图表 | 语法简洁；支持Flowchart/Sequence/State/ER等；社区生态成熟 | Typst中需WASM渲染，编译慢；样式定制有限；中文字体可能有坑；用例图支持极差 | 流程图、ER图、时序图 |
| **Fletcher** (0.5.8) | Typst原生矢量绘图 | 编译快；矢量质量极高；丰富的节点形状/箭头；支持数学公式；中文完美支持；可版本控制 | 复杂图代码量大；需手写坐标 | 架构图、流程图、状态机、网络拓扑 |
| **CeTZ** | Typst原生Canvas | 底层绘图能力最强；完全可控 | 学习曲线陡；代码量大 | 复杂自定义图表 |
| **保留PNG** | 静态位图 | 截图真实；无需修改 | 放大模糊；无法版本控制；修改需重新截图 | UI界面截图 |

## 四、实施要点

### 4.1 包导入方式

在 `template/thesis.typ` 顶部添加：

```typst
// Mermaid 支持（用于流程图、ER图）
#import "@preview/beautiful-mermaid:0.1.0": render as mermaid-render

// Fletcher 支持（用于架构图）
#import "@preview/fletcher:0.5.8" as fletcher: diagram, node, edge
```

### 4.2 imagex 兼容封装

由于 `imagex` 期望接收 `image(...)` 或 `content`，Mermaid/Fletcher 渲染结果本身就是 `content`，可直接传入：

```typst
#imagex(
  #mermaid-render("graph TD; ..."),
  caption: [xxx],
  label-name: "xxx",
)
```

### 4.3 原图备份策略

被替换的 `.png` 文件先保留在 `figures/backup/` 目录中，确认编译无误后再清理。

## 五、预期效果

- **矢量清晰度**：无论放大到多少倍，图表边缘始终锐利，适合打印和电子版阅读
- **版本可控性**：图表源码随论文一起版本控制，修改无需重新导出PNG
- **风格一致性**：所有重绘图表使用统一配色和字体，与论文整体风格协调
- **学术专业性**：ER图、架构图使用标准符号体系，提升论文专业感

## 六、第一轮实施与回滚复盘

### 6.1 第一轮替换（已回滚）

| 序号 | 原图 | 替换为 | 结果 |
|------|------|--------|------|
| 1 | logistics-service-mode.png | Mermaid flowchart (graph LR) | **回滚** — 7节点横向铺开，高宽比极度扁平，文字拥挤 |
| 2 | e-commerce-flow.png | Mermaid flowchart (graph TD) | **回滚** — 9节点纵向排列，高宽比极度细长 |
| 3 | architecture-overview.png | Fletcher diagram | **回滚** — 无分层颜色、节点分布不均、线条交叉混乱 |
| 4 | frontend-architecture.png | Fletcher diagram | **回滚** — 5节点单列纵向，过于细长 |
| 5 | er-diagram.png | Mermaid erDiagram | **回滚** — 19实体自动布局拥挤，无颜色分组，可读性差 |

### 6.2 回滚原因（严厉审查结论）

对比原PNG与Mermaid/Fletcher渲染效果，发现以下致命问题：

1. **长宽比失控**：Mermaid LR/TD 自动布局在A4页面宽度约束下，横向铺开节点导致扁平，纵向排列导致细长。
2. **线条混乱**：Fletcher手动坐标定位的架构图，边连接未经过优化，多条斜线交叉重叠。
3. **缺乏专业元素**：原图有分层颜色背景（架构总览图）、泳道（流程图）、业务域分组（ER图）、特殊形状（云形、圆形）。Mermaid/Fletcher均无法复现。
4. **分辨率不足不能成为理由**：物流服务模式图(501×321)和订单流程图(621×162)原图分辨率确实低，但Mermaid/Fletcher替代方案视觉效果更差，得不偿失。

## 七、工具适用性结论

| 图表类型 | 推荐工具 | 理由 |
|----------|----------|------|
| 复杂分层架构图 | **专业工具 → 导出PNG** | 需要颜色分组、均匀布局，Fletcher代码量过大且难达效果 |
| 带泳道/角色的流程图 | **专业工具 → 导出PNG** | Mermaid无泳道支持，Fletcher手动实现代码量巨大 |
| 复杂ER图（>10实体） | **专业工具 → 导出PNG** | Mermaid ER自动布局在实体多时不可控 |
| 简单流程图（3~5节点） | **Mermaid / Fletcher** | 节点少时自动布局可控，矢量收益明显 |
| 简单状态机 / 转换图 | **Fletcher** | 节点少、边清晰，矢量质量高 |
| 统计图表 | **Cetz / lilaq / 专业工具** | Mermaid完全不支持 |
| UI界面截图 | **保留PNG** | 无法替代 |

## 十、本轮三张图重绘的可复用结论（2026-05-17）

1. **系统架构图 / 前端架构图**：当目标风格接近 draw.io / Visio 的规整分层图时，优先采用脚本直接生成白底高清 PNG，而不是在 Typst 中强行拼复杂布局。此次采用 `PIL` 脚本统一生成边框、圆角、留白和字号，最终效果明显比 `facade`/`fletcher` 试验版稳定。
2. **E-R 图**：对于实体较多的复杂 E-R 图，不要强求单图全塞入一页。更稳妥的方案是按业务域拆成四张子图（用户与角色、商品与库存、订单与配送、调度与网络），分别导出 PNG 后再用脚本组合成总图。
3. **组合方式**：若 Typst 主文档中对子图 PNG 的组合排版不稳定，直接使用外部脚本（如 `PIL`）拼版比在文档内继续调参更可靠。
4. **视觉原则**：论文插图应优先保证低饱和、细边框、白底、统一字号与均匀留白；用户若提供参考图，应先对齐视觉语言，再决定使用什么工具。
