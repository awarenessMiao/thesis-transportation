# 答辩PPT评委攻击点与修改建议

## 适用范围
- 材料来源：`slides/slides.md`、`slides/讲稿.md`
- 使用场景：从“对 AIGC 敏感、会追问证据链和真实性”的评委视角审查毕业答辩内容

## 总体判断
当前 PPT 和讲稿的主要问题不是“不会讲”，而是**很多表述过于完整、过于漂亮、过于像标准答案**，容易让评委产生两种怀疑：
1. **是不是 AI 帮你把项目包装过头了**；
2. **是不是很多结论没有经过真实实验和可重复验证**。

因此修改原则不是继续“写得更强”，而是：
- 少说绝对化结论；
- 多给可验证边界；
- 把“创新”降级为“尝试/改进/工程实践”；
- 把“系统都做完了”改成“完成了哪些、验证了哪些、还有哪些局限”。

## 最高风险攻击点

### 1. 研究背景与研究现状像“AI 概括”，缺乏来源和落点
高风险表述：
- “物流成本占 GDP 比重 14.1%”
- “国内外研究不足”整页几乎全是结论句
- “国外缺 LLM 工程实践”

评委可能攻击：
- 这些数字和文献来自哪里？
- 你说别人“缺工程落地”，你看了几篇？怎么归纳的？
- 你的研究现状是文献综述，还是 AI 帮你总结的行业常识？

修改建议：
- 每个关键统计口径都补来源，答辩口头要能说出“国家统计/行业报告/具体论文”。
- 不要一页直接判定“国内外不足”，改成“选取若干代表工作，对比其关注点与本文工作范围”。
- 把“国外缺 LLM 工程实践”改成“在本课题检索范围内，公开可见的物流路径规划与微服务结合的 LLM 工程案例较少”。

### 2. “创新点”说得过满，容易被打成工程拼装
高风险表述：
- “核心创新点”
- “LLM+确定性算法混合决策新模式”
- “Hub-and-Spoke + MCMF 工程实践”
- “策略模式运行时切换”也被放进创新点

评委可能攻击：
- 策略模式是常规软件设计，为什么算创新？
- 你这个所谓混合决策，是否只是“调用了一个大模型 API”？
- K-Means、A*、MCMF 都是成熟算法，创新到底在哪里？

修改建议：
- 把“创新点”统一降级成“本文工作特点”或“本文的三点改进”。
- 三点改成：
  1. 在物流微服务系统中实现了可切换的路径规划策略框架；
  2. 尝试将 LLM 限制在“裁决/路点建议”角色，并设置规则校验与降级机制；
  3. 将 K-Means 与 MCMF 接入实际业务流程界面进行工程验证。
- 口头上避免说“理论创新”，改说“系统集成创新”或“工程实现上的组合式改进”。

### 3. LLM 描述非常招黑，容易被质疑“为了蹭热点硬加 AI”
高风险表述：
- “避免 LLM 幻觉”
- “LLM 战略层 + A* 战术层”
- “LLM 费率校准”
- “LLM 辅助分析簇间合并、优先级”

评委可能攻击：
- 你的 LLM 输入是什么？输出是什么？怎么评估它真的提升了效果？
- 如果没有 LLM，这个系统是不是照样能跑？
- 既然路线坐标还是 A* 出的，那 LLM 的必要性是什么？
- “费率校准”有没有真实业务数据支撑？还是概念性描述？

修改建议：
- 严格区分“已落地实现”“实验性尝试”“可选扩展”。
- 如果没有扎实实验，不要在 PPT 正文里讲“LLM 费率校准”和“K-Means 的 LLM 辅助分析”，可移到“展望/可扩展点”。
- “避免幻觉”改成“通过约束输出范围、地理校验和失败降级，降低不可控输出对系统的影响”。
- 口头必须承认：**系统核心可在不依赖 LLM 的情况下运行，LLM 是增强模块，不是基础前提。**

### 4. 测试与结果缺少实验设计，容易被认定“截图式验证”
高风险表述：
- “✅ 全部核心功能测试通过”
- “✅ 三种策略基准测试结果”
- “✅ MCMF 运力规划 — 68 条订单 + Hub 网络”
- “✅ Docker Compose 一键部署”

评委可能攻击：
- 测试环境、测试集、评价指标是什么？
- 你说基准测试，baseline 是谁？
- 你说 MCMF 有结果，如何验证优于人工规则或贪心策略？
- 你这是功能演示，不等于实验结论。

修改建议：
- 把“测试概要”页改成“三类验证”：功能可用性验证、算法行为验证、部署可复现性验证。
- 不要写满屏“✅”，改成表格：验证对象 / 验证方法 / 结果 / 局限。
- 对路径规划补充最基本四件事：测试 OD 数量、每种策略运行次数、评价指标、结论范围。
- 如果没有充分对照实验，就把“基准测试结果”改成“样例对比结果”或“实验性对比结果”。

### 5. 部署表述太理想化，现场一追细节就容易露怯
高风险表述：
- “一键 docker compose up -d 启动”
- “数据库自动初始化 + 种子数据”
- “ARM64 兼容”

评委可能攻击：
- 你是否在一台新机器上完整复现过？
- 所有数据都能自动导入吗？还是需要补脚本？
- Nacos、MySQL、RabbitMQ 的就绪顺序怎么处理？

修改建议：
- PPT 中把结论改保守：
  - “系统支持基于 Docker Compose 的整体部署”；
  - “基础环境可自动初始化，演示数据通过初始化脚本补充”；
  - “已在 Apple Silicon 环境完成兼容性验证”。
- 如果现场不打算演示部署，千万不要主动把自己推到“可完全复现”这个战场上。

### 6. 太多“标准化漂亮术语”，但缺少你自己的取舍与失败经验
评委敏感点：
- 一听就像 AI 写的顺滑表达：
  - “兼顾智能性与可靠性”
  - “促进行业数字化转型”
  - “工程化落地”
  - “全链路协同”

修改建议：
- 每章至少加一句**你自己的真实取舍**：
  - 为什么没有做实时路况？
  - 为什么 LLM 只保留为裁判而不是直接生成路线？
  - 为什么 MCMF 只做静态批量规划而不是在线调度？
- 评委往往更相信“知道自己没做什么”的学生，而不是“什么都做了”的学生。

## 建议直接修改的页面

### 第3-4页：背景与现状
- 删除或弱化没有来源的绝对数字和行业判断。
- 补 2~3 篇最关键参考工作名称或作者年份。
- 口头表达从“行业判断”改为“文献归纳”。

### 第5页：研究内容
- 删掉“智能算法设计”卡片里过度展开的术语堆叠。
- 用“路径规划增强、末端聚类、运力规划”代替一长串算法名。

### 第15-19页：智能算法部分
- 保留 `A* / LLM Judge / LLM Waypoint / K-Means / MCMF` 主线。
- 把“LLM费率校准”“LLM辅助分析簇间合并、优先级”移到“可扩展方向”或干脆删除。
- 每页增加“输入、输出、约束、失败回退”四个维度中的至少两个。

### 第20页：基准测试
- 若没有严格实验设计，改名为“样例实验对比”。
- 页脚补上“测试范围有限，主要用于验证策略行为差异，不构成普适统计结论”。

### 第21页：测试概要
- 删除大面积 `✅` 风格。
- 改为“验证内容 / 说明 / 结论 / 局限”表格，可信度会明显提升。

### 第23页：总结与创新点
- 标题从“创新点”改为“工作特点与贡献”。
- “策略模式运行时切换”从创新点里拿掉，放到“工程实现特点”。
- 所有“实现了”“证明了”尽量改成“完成了”“验证了可行性”“进行了初步探索”。

## 高频答辩 Q&A（按高概率排序）

### Q1：你这个课题真正的创新点是什么？
建议回答：
“如果从学术理论创新来讲，我的工作更多是**工程集成与方案改进**，不是提出了全新的基础算法。我的主要贡献有三点：第一，把物流业务拆成微服务并完成了核心流程实现；第二，在路径规划中尝试把 LLM 限制在裁决或路点建议角色，并通过规则校验和降级机制控制风险；第三，把 K-Means 和 MCMF 接入了实际业务流程，完成了可运行的系统验证。”

### Q2：为什么一定要用 LLM？不用 LLM 系统能不能跑？
建议回答：
“能跑。系统的基础能力并不依赖 LLM，A*、K-Means、MCMF 都可以独立工作。引入 LLM 的目的不是替代确定性算法，而是做受限的增强决策，比如在多个候选路径之间做语义化裁决，或者给出可校验的中间路点建议。因此我把它设计成可关闭、可降级的可选模块。”

### Q3：你怎么证明 LLM 真的带来了提升？
建议回答：
“这部分我目前只能说做了**初步样例验证**，还不足以形成大规模统计结论。我的重点是证明这种混合架构在系统层面可实现、可降级、可约束。若继续深入，我会补充更大规模的 OD 数据集、对照实验和更明确的评价指标。”

### Q4：既然路线还是 A* 算出来的，那 LLM 有什么价值？
建议回答：
“价值不在于替代底层寻路，而在于参与更高层的方案选择。A* 擅长在给定代价模型下求路径，但对于‘绕开可能拥堵区域’‘在多个接近方案中结合历史经验做取舍’这类问题，LLM 可以作为有限的上层决策器。不过目前我也承认，这种价值更多体现在探索性增强，而不是已经被严格证明的性能优势。”

### Q5：你的研究现状和文献综述是不是 AI 帮你写的？
建议回答：
“我在整理时确实使用了工具辅助归纳表述，但最终保留的观点都回到了论文和项目本身来核对。答辩时我更愿意把研究现状说成‘几类代表工作的对比’，而不是泛泛地下结论。后续我也会把关键统计和文献来源在 PPT 中标得更清楚。”

### Q6：策略模式为什么能算创新？
建议回答：
“严格说它不属于理论创新，更适合作为工程设计特点。它的价值是把三种路径规划策略解耦，便于比较、切换和扩展。所以如果表述不严谨，我会把它从‘创新点’调整为‘系统实现特点’。”

### Q7：MCMF 这一块和真实物流业务之间差距有多大？
建议回答：
“差距还是有的。当前模型主要面向静态批量规划，简化了实时车况、司机班次、突发事件、跨天约束等现实因素。我的工作重点是先验证 Hub-and-Spoke 网络下 MCMF 在系统中的可接入性和基本规划能力，而不是完整复刻真实物流企业的调度复杂度。”

### Q8：K-Means 为什么取 \(k=\lceil\sqrt{n}\rceil\)？有依据吗？
建议回答：
“这是一个工程启发式取值，用来快速给出可演示的聚类结果，并不是理论最优选择。实际部署中，\(k\) 应该结合司机数量、容量约束、服务时窗等业务条件共同确定。论文里我更应该把它表述为‘经验设定’，而不是通用最优策略。”

### Q9：你的测试是不是只是在展示页面截图？
建议回答：
“确实有一部分是功能演示截图，这只能说明系统可运行，不能直接等同于严格实验。因此我会把测试结果区分成三类：功能可用性验证、算法行为样例验证和部署验证，避免把它们混成一个‘全部证明完成’的结论。”

### Q10：如果让你删掉一个最不稳的点，你会删什么？
建议回答：
“我会优先弱化或删除没有充分实验支撑的 LLM 扩展性表述，比如费率校准、过强的性能优越性结论，保留真正完成实现和验证的核心部分。这样整体可信度会更高。”

## 一句话总策略
**把“我做了一个很厉害的 AI 物流系统”改成“我完成了一个可运行的微服务物流系统，并对 LLM 受限增强决策做了初步探索”。**

这句话一改，整套材料的可信度会明显上升。
